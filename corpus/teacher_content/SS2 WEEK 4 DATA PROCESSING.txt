Normal Form
Normal Form is a set of rules and guidelines used in database design to reduce data redundancy and ensure data integrity. The process of organizing data into normal forms is known as Normalization. Normalization helps in structuring the database in a way that minimizes duplication of data, ensures dependencies are logical, and optimizes data retrieval.
Problems Facing Normal Form
While Normalization is essential for efficient database design, it also comes with certain challenges:
Complexity: Moving data into higher normal forms (such as 3NF or beyond) can result in increased complexity, as it requires splitting tables into multiple smaller ones, which may make the database harder to understand and manage.
Performance Issues:  Highly normalized databases may require more complex queries with multiple table joins. This can lead to slower performance in read-heavy operations since retrieving data from multiple tables can be resource-intensive.
Over-normalization: In some cases, excessive normalization can cause unnecessary splitting of data, leading to too many small tables and increased difficulty in managing the database.
De-normalization Requirements: Sometimes, denormalization (combining tables back into larger ones) is necessary to improve performance in real-world applications, especially in systems that require fast data retrieval.

Examples of Tables in First, Second, and Third Normal Form
1.  First Normal Form (1NF): A table is in 1NF if all the columns contain atomic (indivisible) values, and each entry in the column must contain only one value.
Example:
2.  Second Normal Form (2NF):
A table is in 2NF if it is in 1NF and all non-key attributes are fully dependent on the entire primary key (no partial dependency on part of the primary key).
Example
3. Third Normal Form (3NF)
A table is in 3NF if it is in 2NF and no transitive dependency exists (non-key attributes are not dependent on other non-key attributes).
