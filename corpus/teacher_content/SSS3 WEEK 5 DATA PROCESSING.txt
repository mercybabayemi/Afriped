Definition:
Crash recovery is the process of restoring a database to a consistent and usable state after a system failure. This involves rolling back incomplete transactions and completing committed transactions that were still in memory when the crash occurred. A database is considered to be in a consistent state when it has reached a point of consistency.
Conditions that can result in transaction failure:
1. Power failure: A sudden loss of power can cause the database manager and its partitions to go down.
2. Hardware failure: Issues with memory, disk, CPU, or network components can lead to database failures.
3. Operating system errors: Serious errors in the operating system can cause the database to crash.
THE LOG
The Log, sometimes called Trail or journal, is a series of actions executed by the DBMS. Physically, the log is a file of records stored in a stable storage, which is assumed to survive crashes. Every log record  is given a unique ID called the Log Sequence Number.

Introduction to ARIES (Algorithms for Recovery and Isolation Exploiting Semantics)
ARIES is a recovery algorithm designed to work with a no-force, steal database approach. It is used by IBM DB2, MS SQL Server, and many other database systems.
The three main principles that underlie the ARIES recovery algorithm are:
1. Write-Ahead Logging: Any change to an object is first recorded in the log, and then the log must be written to stable storage before changes to the object are written to a disk.
2. Repeating History during Redo: On restart after a crash, ARIES retraces the actions of a database before the crash to bring the system back to the exact state it was in before the crash. Then, it undoes the transactions still active at crash time.
3. Logging Changes during Undo: Changes made to the database while undoing transactions are logged to ensure that such actions aren't repeated in the event of repeated restarts.
